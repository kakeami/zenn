## 背景

### 「データ可視化技術入門」とは

「データ可視化技術入門」（以降、「本書」と呼びます）は、マンガ・アニメ・ゲームのデータを使って、楽しくデータ可視化の基礎を学ぶ入門書です。上巻と下巻の分冊構成となっており、それぞれ基本と応用に焦点を当てています。

https://gihyo.jp/book/2026/978-4-297-15271-0

https://gihyo.jp/book/2026/978-4-297-15273-4

### なぜ執筆時間を可視化するか

理由は主に3つあります。

1. **自分自身のため**：私を支えてくれた家族（とくに妻）への感謝を忘れないために。また、集中力のない自分をどう操縦すれば作業を進められるのか、そのヒントを得たいと思いました。

2. **未来の技術書執筆者のため**：技術書を執筆する度に「こんなに大変だったとは」と思います。執筆時間に関する**定量的な記録**があまり見当たらないことが一因ではないでしょうか。私は2019年頃から全ての作業時間を秒単位で記録しており、これを公開することで参考になればと考えました。

3. **本書の読者のため**：本書で紹介する技術の活用例として、執筆時間を分析・可視化しています。以下は本サイトと本書の対応関係を表します。

|  | 本サイトで扱う内容 | 本書の該当箇所 |
| :--- | :--- | :--- |
| 1 | [データの記録](https://kakeami.github.io/road-to-vizbook/01/track)| 上巻7章|
| 2 | [データの取得](https://kakeami.github.io/road-to-vizbook/01/download)| 上巻6章、上巻7章|
| 3 | [データの前処理](https://kakeami.github.io/road-to-vizbook/01/preprocess)| 上巻6章|
| 4 | [データの基礎分析](https://kakeami.github.io/road-to-vizbook/01/eda)| 上巻6章|
| 5 | [量を見るためのデータ可視化](https://kakeami.github.io/road-to-vizbook/02/amounts)| 下巻1章 |
| 6 | [分布を見るためのデータ可視化](https://kakeami.github.io/road-to-vizbook/02/dists) | 下巻2章 |
| 7 | [内訳を見るためのデータ可視化](https://kakeami.github.io/road-to-vizbook/02/props) | 下巻3章 |
| 8 | [関係を見るためのデータ可視化](https://kakeami.github.io/road-to-vizbook/02/assocs) | 下巻4章 |

### 前作との比較

本サイトは、5年前以上前に作成した「[ジョージア工科大学に合格するまでの572時間29分02秒](https://kakeami.github.io/road-to-gatech/)」の続編にあたります。


![](https://storage.googleapis.com/zenn-user-upload/eguvx8d7rm2r4ch2krvo3uv31dcp)

前作では、ジョージア工科大学大学院のオンラインプログラムに合格するまでの準備時間（**572時間45分17秒**）を可視化しました。以下は今作と前作の比較表です。

| 項目 | [前作（road-to-gatech）](https://kakeami.github.io/road-to-gatech/) | [今作（road-to-vizbook）](https://kakeami.github.io/road-to-vizbook/) |
| :--- | :--- | :--- |
| 対象 | 留学準備 | 技術書執筆 |
| 分析対象時間 | 572時間45分17秒 | 1161時間50分38秒 |
| サイト作成時間 | 82時間13分29秒 | 84時間00分26秒 |
| 静的サイトジェネレータ | MkDocs | Jupyter Book |

対象作業の時間は約2倍になりましたが、サイトの作成時間はほぼ同じでした。全くの偶然ですが、興味深いですね。

静的サイトジェネレータとして、今回は[Jupyter Book](https://jupyterbook.org/v1/intro.html)を採用しました。選定理由は後述します。

ちなみに今作は、**書籍で紹介するデータ可視化技術で、その書籍自体を分析する** というメタ構造になっている点がポイントです。

## 作成方法

### データの記録

[Todoist](https://todoist.com/ja)のタスクを[Chrome拡張機能](https://todoist.com/ja/help/articles/use-toggl-track-with-todoist)を使って[Toggl track](https://toggl.com/track/)と連携することで、作業時間を記録しました。可能な限り実態に近づけるため、**トイレ休憩等のたびに計測を止めました**。

![](/images/2026-01-vizbook/toggl.png)
*ある週の記録結果（執筆以外も含まれるため、タスク名は黒塗り）*


### データの取得

[Toggl API v9](https://engineering.toggl.com/docs/)を使ってデータを取得しました。Rate Limitがあるため、制限に達した場合は自動的に待機してリトライする処理を実装しています。

一つ工夫した点として、日付の基準を「開始時刻」ではなく「終了時刻」にしました。これは睡眠時間の日付ズレを回避するためです。例えば深夜1時まで作業して就寝した場合、開始時刻基準だと前日の睡眠時間が極端に短くなってしまいます。

![](/images/2026-01-vizbook/end_date.png)
*開始時刻の日付（赤）と終了時刻の日付（青）。本サイトでは青を採用した*

### データの可視化

[Plotly](https://plotly.com/python/)を使用しています。（質的変数の）カラーパレットには[Okabe-Itoパレット](https://jfly.uni-koeln.de/color/)を採用しました。これは色覚の多様性に配慮した配色で、本書でも推奨しているパレットです。

![](/images/2026-01-vizbook/oi.png)
*Okabe-Itoカラー*

### サイトの構築

[Jupyter Book](https://jupyterbook.org/)を使用しています。前作ではMkDocsを使用していましたが、今回は以下の理由でJupyter Bookを選びました：

- Jupyter Notebookとの親和性が高いため
- コードと出力を一緒に管理できるため
- 本書の[サポートサイト](https://kakeami.github.io/vizbook-jupyter/)と技術スタックを統一できるため

GitHub Pagesでホスティングしています。

## 可視化結果

### カテゴリー別・章別の作業時間

サイトでは、棒グラフやヒートマップなど複数の手法で作業時間を可視化しています。以下は、カテゴリー別の合計作業時間です。

| カテゴリー | 作業時間 | 内容 |
| :--- | ---: | :--- |
| 原稿執筆 | 約900時間 | データ準備、可視化、草稿執筆など |
| 校正校閲 | 約108時間 | 編集者との内容・構成・文章表現の見直し |
| プロトタイピング | 約75時間 | 前身サイトの設計・実装・公開 |
| 企画立案 | 約50時間 | 目次確定、コンセプト詰め |
| 販促 | 約30時間 | 練習問題作成、SNS告知など |

**原稿執筆が全体の約8割**を占めています。特に時間がかかったのは以下のタスクでした：

- **上巻3-4章**（約142時間）：マンガデータを用いたハンズオン
- **上巻1-2章**（約115時間）：データ可視化の教科書的知識の総ざらい
- **下巻5-6章**（約106時間）：メディア展開データを用いたハンズオン

章ごとに分解できていないのは、執筆中に構成が大きく変わったためです。

### 月別・曜日別の作業パターン

ヒートマップで月別・曜日別の作業時間を可視化すると、興味深いパターンが見えてきました。

- **2022年1月**：曜日を問わず作業時間を確保できていた（プロトタイピング期間）
- **2023年11月〜2024年4月**：土日を中心に作業時間が急増（原稿執筆の追い込み期間）
- **2024年後半〜**：校正校閲フェーズに移行

想定通り、平日より土日の作業時間が長い傾向がありました。平日の中では金曜が最も長く、これは「翌日が休みなので夜に無理をしやすい」という私のライフスタイルを反映しています。

### 作業時間の分布

作業1回あたりの継続時間を分析すると：

- **全体の約1/4**は30分以下の短時間作業
- **最頻値**は10分程度の作業
- **最長記録**は3時間超（コンテスト発表準備）

「日々の小さな積み重ねが大きな成果につながる」ことが数字で確認できました。


## 苦労したこと

未来の技術書執筆者のために、誤算や苦労話も正直に共有します。

### プロトタイピングの誤算

本書の前身として、[マンガと学ぶデータビジュアライゼーション](https://kakeami.github.io/viz-madb/index.html)というサイトを公開していました。コンテストで発表し、SNSでも批判的なコメントは見当たらなかったため、「あとはこの内容を**少しだけ**補強すれば書籍化できる」と想像していました。

しかし、この想定は甘かったです。企画立案フェーズで編集者と議論した結果：

- マンガだけでなく、**アニメやゲームも扱う**
- MADBだけでなく、**外部のデータソースも扱う**
- それらを組み合わせた**発展的な分析**を行う

となりました。もう別物です。ですが、このブラッシュアップのおかげで、今の形で本書を世に送り出すことができました。

### 上巻3-4章の苦労

上巻3-4章はマンガデータを用いたハンズオンです。プロトタイプと重複しているため、早々に片付ける想定でした。

しかし、プロトタイプを改めて見返してみると、可視化例にムラが目立ちました。特に後半部分は「だから何？」といった可視化結果が並び、本として一貫したストーリーを語りづらいと感じました。そこで、泣く泣く一から可視化例を練り直すことになります。

解決策として、仮説ドリブンのデータ可視化を採用しました。物語を推進するドライバーとして**Aさん**と**N博士**というキャラクターを考案し、二人のやりとりを通じて本としての体をなすようになりました（と信じています）。

### 校正校閲の大変さ

校正校閲は想像以上に大変でした。

本書は、いたるところにキャラクター同士の会話が挟まる特殊なレイアウトを採用しています。本文・図・ソースコードだけでなく、全12種類ずつあるキャラクターのパターン、本文と連携した挿入位置などの確認が必要になりました。

おまけに本書は上下巻合わせて約800ページの大ボリュームです。1ページを1分で確認できたとしても、全ページチェックするのに13時間以上かかります。実際には修正箇所にコメントを入れたり、画像を作り直したりするため、**1ページを1分で確認することは不可能**でした。

### 予想外の追加作業

Jupyter Book版の草稿が完成した時点で、「あとはマークダウンに清書するだけ」と思っていました。しかし実際には、以下のような修正が必要になりました：

- **分冊化**：分量の問題から分冊化することに。上巻と下巻を接続する**上巻7章**を新たに執筆する作業が発生
- **会話の追加**：文字ばかりで読みづらくなることを避けるため、AさんとN博士の会話文を追加。最終的に**945の会話文**を追加

また、執筆中にダイナミックに章構成が変わったことで、**Togglのトラッキングデータの粒度が実際の章構成と合わなくなる**という問題も発生しました。当初のタスク設計では章単位での時間計測を想定していましたが、章の分割・統合・追加が繰り返されたため、「上巻1-2章」「上巻3-4章」「下巻5-6章」のように複数章がまとめて記録されてしまいました。

## わかったこと

### 「10分でいいからやる」の効果

分析を通じて、作業時間の約1/4は30分以下の短時間作業で構成されていることがわかりました。作業回数では10分程度の作業が最も多かったです。

こと私という人間の操縦においては、「やる気」が起きるのを待つのは得策ではないようです。そもそも私は怠惰なので、やる気に溢れた日は一年の中で数える程度しかありません。重要なのは**如何に自分を騙し、初動の心理的ハードルを下げるか**だと感じています。「10分でいいからやる」を座右の銘にしたいと思います。

### 技術書執筆のコスト感

本書は上下巻合わせて約80万文字で構成されています。これに約1160時間かかりましたので、単純計算で**1万文字あたり約14.5時間**かかる計算になります。

本書がやや特殊な作りである点は注意が必要です：

- マンガ・アニメ・ゲームデータという特殊なテーマ
- 可視化と分析のコードをほぼすべて自作
- キャラクター（AさんとN博士）による対話形式（合計945の会話文）
- 分冊化による構成の複雑化

しかし、それでも一つの基準にはなるのではないかと思います。参考になれば幸いです。

### 両輪で回す

まとまった作業時間を確保することの重要性も明らかになりました。2024年になって30分以上の作業時間が飛躍的に増加し、この時期に本書の多くの部分が完成しました。

「小さく前進し続ける」戦略と「集中的に進める」戦略の**両輪で回す**ことがポイントだったのかもしれません。

### 思い通りにいくとは限らない

誤算と苦労話で述べた通り、技術書執筆は思い通りにいくとは限りません。「少しだけ補強すれば」が別物になり、「早々に片付ける」章に最も時間がかかり、「清書するだけ」のはずが大幅な追加作業が発生しました。

振り返ると、これらの誤算は全て**良い方向への軌道修正**でした。編集者との議論を通じて、当初の想定より遥かに良い本になったと感じています。誤算を恐れず、柔軟に対応することが大切なのかもしれません。

### タスク管理は細かい粒度で

技術的な教訓として、**タスク管理・時間記録は可能な限り細かい粒度で行うべき**ということを学びました。

今回、章単位でタスクを設計していたため、執筆中に章構成が変わった際にトラッキングデータの粒度が実際の構成と合わなくなりました。もし節や項の単位で記録していれば、章の分割・統合があっても柔軟に集計し直せたはずです。

長期プロジェクトでは構成変更は珍しくありません。将来の分析の柔軟性を確保するためにも、**最初から細かい粒度で記録しておく**ことをお勧めします。

## おわりに

1161時間50分38秒は長かったですが、楽しかったです。

分析を通して新たな発見がいくつもあり、これまで以上にマンガ、アニメ、そしてゲームを好きになりました。これは本書の中でも何度か述べた「好きなものを分析すると学びが深まる」という主張を、私自身が体現した形になりました。

改めて、作業環境を支えてくれた家族には心から感謝しています。特に妻の理解と支援がなければ、この本が完成することはありませんでした。まずは家族に恩返しをしたいです。

最後に、**好きなもの・興味のあるものを巻き込むと、学習がはかどります。** マンガやアニメに興味がない方は、このサイトで紹介するように**自分自身**を対象にデータ可視化をしてみてはいかがでしょうか？
